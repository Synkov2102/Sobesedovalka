# Бэкенд: соглашения по стилю и структуре

Краткий гайд для NestJS-проекта. Цель — предсказуемая структура, строгая типизация и единообразный HTTP/API.

## Архитектура и модули

- **Один домен — один модуль** (`TasksModule`, `CollabModule`, `HealthModule`). В `AppModule` только импорты модулей, без лишней логики.
- **Слои внутри фичи:**
  - `*.module.ts` — декларация модуля;
  - `*.controller.ts` — маршруты HTTP (тонкий слой);
  - `*.service.ts` / `*.gateway.ts` — бизнес-логика и интеграции;
  - `dto/` — входящие тела и query (с валидацией);
  - `interfaces/` или `entities/` — модели данных, не привязанные к транспорту.
- **Транспорт:** REST отдельно от WebSocket; общие типы домена выносить в `*.types.ts`, а не дублировать в gateway.

## HTTP и валидация

- Глобальный **`ValidationPipe`**: `whitelist`, `forbidNonWhitelisted`, `transform` — неизвестные поля отсекаются, типы приводятся к DTO.
- Входящие тела оформлять **классами DTO** + `class-validator` / `class-transformer`, а не «сырыми» `{ title?: string }` в контроллере.
- Идентификаторы с известным форматом валидировать пайпами Nest (`ParseUUIDPipe` и т.д.).
- Успешное удаление — **`204 No Content`**, если тела ответа нет.

## Код-стайл

- **Prettier** — единый формат; не спорить с автоформатом в CI/IDE.
- **ESLint** + `typescript-eslint`: типобезопасные правила; **`consistent-type-imports`** — отдельные `import type` для типов.
- **TypeScript `strict: true`** — включать и исправлять ошибки, а не отключать проверки.
- Поля, инициализируемые декораторами Nest (`@WebSocketServer()` и т.п.), помечать **`!`** при необходимости для `strictPropertyInitialization`.

## Bootstrap и инфраструктура

- **`bootstrap().catch(...)`** — логирование и `process.exit(1)` при падении старта.
- Префикс API (`api`), CORS и порт — централизованно в `main.ts`; для продакшена лучше вынести в `@nestjs/config` и переменные окружения.

## Тесты

- Юнит-тесты рядом с фичей или в `*.spec.ts`; контроллеры покрывать через `TestingModule` без поднятия HTTP, если достаточно вызова методов.

## Именование

- Файлы в **kebab-case** (`create-task.dto.ts`, `task.interface.ts`).
- Классы — **PascalCase**, методы/поля — **camelCase**.
- Сообщения об ошибках для клиента — на английском или едином языке продукта, но последовательно.

---

При добавлении новой фичи: сгенерировать модуль Nest CLI или скопировать структуру существующей фичи (`tasks/` или `health/`).
